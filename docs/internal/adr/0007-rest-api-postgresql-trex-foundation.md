# ADR-0007: REST API + PostgreSQL Foundation via rh-trex-ai

**Status:** Accepted
**Date:** 2026-02-16
**Deciders:** Mark Turansky, Kevin Howell
**Technical Story:** [GitHub Issue #642](https://github.com/ambient-code/platform/issues/642) · [Technical Evaluation Gist](https://gist.github.com/markturansky/5ca6c955b536f344c24af5b90f5f07bb)

## Context and Problem Statement

The original platform (ADR-0001) stored all business data — sessions, projects, settings — as Kubernetes Custom Resources in etcd. While this worked for initial delivery, three problems emerged:

1. **etcd is not a relational database.** Querying, indexing, and relational joins across CRDs require K8s list operations that do not scale.
2. **No API without Kubernetes.** The only external API was `public-api`, a stateless proxy that forwarded requests to the backend, which then talked to the Kubernetes API server. External consumers had an implicit dependency on Kubernetes internals.
3. **Model drift.** Type definitions were duplicated: Go structs in the backend, TypeScript types in the frontend, and DTO shapes in public-api — with no single source of truth.

A proof-of-concept migration tool (`feature/cdr-to-database`) was created to export CRDs to PostgreSQL. Evaluation confirmed the POC exported data correctly but did not replace the API layer — the backend remained 100% Kubernetes-native. A proper solution required a new, PostgreSQL-native API with generated SDKs and a control plane to reconcile API state back into Kubernetes for execution.

## Decision Drivers

- **Scalability:** PostgreSQL handles relational queries, joins, and indexes at scale; etcd does not
- **Single source of truth:** One OpenAPI spec should generate all language SDKs and eliminate model drift
- **Build on existing investment:** An API framework already exists inside Red Hat that underpins services behind `api.openshift.com`
- **Avoid reinventing boilerplate:** OIDC, DB migrations, server mux, RBAC, event-driven controllers — every API needs these; TRex already has them
- **Generativity:** Code generators should handle boilerplate; differentiated business logic should live in Ambient
- **CLI familiarity:** Generated CLIs should follow the `oc` / `ocm` / `rosa` verb-noun pattern that Red Hat engineers and customers already know
- **Future extensibility:** New cross-cutting capabilities (e.g., gRPC watch streams) belong in the framework upstream so every future API gets them automatically

## Considered Options

### Option 1: Extend the existing backend (Gin + Kubernetes)

Continue using the existing `components/backend` Gin server. Add PostgreSQL as a secondary store alongside etcd. Keep CRDs as the primary resource model.

**Problems:** Two persistence layers to keep in sync. The backend's architecture is tightly coupled to Kubernetes client-go patterns. No path to generating SDKs or CLIs from the API shape. Model drift continues.

### Option 2: Build a new REST API from scratch (Go, any framework)

Write a new Go HTTP service with PostgreSQL using standard library or a lightweight framework. Implement OIDC, migrations, RBAC, search, pagination, event-driven controllers, health, and metrics from scratch.

**Problems:** All of the functionality required already exists in TRex. A technical evaluation concluded the ambient-api-server (rh-trex-ai) was "the stronger technical foundation by a significant margin" over a scratch-built POC. Rebuilding this would take months and introduce equivalent risk without the years of production hardening TRex has behind `api.openshift.com`.

### Option 3: Adopt rh-trex-ai as the API framework (chosen)

Use `rh-trex-ai` as the upstream library and framework. Business logic lives in per-resource plugins generated by TRex's code generators. Boilerplate (OIDC, DB, migrations, routing, event controllers, health, metrics) is inherited from upstream. OpenAPI specs are emitted per plugin and consumed by SDK generators. The control plane bridges the API server to Kubernetes for session execution.

## Decision Outcome

Chosen option: **"Adopt rh-trex-ai as the API framework"**

The decisive factors:

- TRex already exists, is battle-tested at scale behind `api.openshift.com`, and has years of accumulated Red Hat engineering
- The code generator pattern (`--kind Foo --fields ...`) turns a data model definition into a complete, tested CRUD plugin in one command — boilerplate goes upstream, differentiated value stays in Ambient
- The emitted OpenAPI spec is the single source of truth from which Go, Python, and TypeScript SDKs are generated, eliminating model drift structurally
- gRPC watch streaming was added to TRex as a new generator template — every future API built on TRex automatically gets watches, and the Control Plane uses them for real-time reconciliation
- The CLI (`acpctl`) is modeled on `oc` / `ocm` / `rosa` verb-noun patterns, giving Red Hat engineers and customers a familiar interface

### Consequences

**Positive:**

- PostgreSQL as the primary data store — proper indexing, joins, and query scalability
- Single OpenAPI spec → Go SDK + Python SDK + TypeScript SDK — drift is structurally impossible
- TRex generators eliminate boilerplate: one command creates model, handlers, DAO, service, migration, presenter, and tests
- gRPC watch streams in the framework mean every future resource type gets real-time change propagation
- `acpctl` verb-noun CLI pattern is immediately familiar to Red Hat / OpenShift practitioners
- Control Plane reconciler interface is pluggable — Kubernetes reconciliation and in-memory counters (for metrics and testing) are both valid implementations
- Additive rollout: V1 (Kubernetes-native) and V2 (REST/PostgreSQL) run in parallel; no breaking changes

**Negative:**

- rh-trex-ai is an upstream dependency — bugs or API changes in the framework affect Ambient
- Developers must learn the TRex plugin anatomy (plugin.go, model.go, handler.go, dao.go, service.go, presenter.go, migration.go)
- Two API paths (V1 backend + V2 API server) must be maintained during transition
- Control Plane adds an operational component between the API server and Kubernetes that did not previously exist

**Risks:**

- Framework coupling: significant architectural changes in rh-trex-ai may require Ambient to adapt
- Write-back echo: Control Plane must detect and suppress its own status updates to prevent reconciliation loops (mitigated by `UpdatedAt` timestamp comparison)

## Implementation Notes

The platform now has two parallel data planes:

```
V1 (legacy):  Frontend → Backend (Gin) → Kubernetes CRDs → Operator → Runner
V2 (current): Frontend → ambient-api-server (TRex) → PostgreSQL
                          ↕ gRPC watch
              Control Plane → Kubernetes CRDs → Operator → Runner
```

The Control Plane implements a `Reconciler` interface with three production implementations:

| Reconciler | Mode | Purpose |
|------------|------|---------|
| `SessionReconciler` | kube | Creates/patches AgenticSession CRDs |
| `ProjectReconciler` | kube | Creates Namespaces |
| `ProjectSettingsReconciler` | kube | Creates RoleBindings |
| `LocalSessionReconciler` | local | Spawns runner processes directly |
| `TallyReconciler` | test | Counts events; no side effects |

New resource types ("Kinds") are added via the TRex generator:

```bash
go run ./scripts/generator.go \
  --kind MyResource \
  --fields "name:string:required,status:string" \
  --project ambient-api-server \
  --repo github.com/ambient-code/platform/components
```

This produces a complete plugin under `plugins/my_resources/` and a corresponding OpenAPI YAML fragment. Running `make generate` rebuilds the Go client; SDK codegen produces the language SDKs.

**Key Files:**

- `components/ambient-api-server/` — TRex-based REST + gRPC API server
- `components/ambient-api-server/plugins/` — one plugin per resource Kind
- `components/ambient-api-server/openapi/` — OpenAPI YAML specs (source of truth)
- `components/ambient-api-server/scripts/generator.go` — Kind code generator
- `components/ambient-sdk/` — Go, Python, TypeScript SDKs generated from OpenAPI
- `components/ambient-control-plane/internal/reconciler/reconciler.go` — Reconciler interface + implementations
- `components/ambient-control-plane/internal/informer/informer.go` — list-sync + gRPC watch engine
- `components/ambient-cli/` — `acpctl` CLI built on the Go SDK

**Patterns Established:**

- All new resource types use the TRex plugin generator — no hand-written CRUD
- OpenAPI spec is the single source of truth; SDK types are never written manually
- Control plane reconcilers implement an interface — Kubernetes is one target, not the only one
- gRPC watch streams handle real-time propagation; the REST API handles CRUD
- Write-back echo is suppressed via `UpdatedAt` timestamp comparison in the informer

## Validation

- All three SDKs (Go, Python, TypeScript) build clean from the same `openapi.yaml`
- 112 SDK tests pass (43 TypeScript + 69 Python)
- Control Plane list-sync + gRPC watch functional against live API server
- `acpctl` can create, list, describe, start, and stop sessions against the V2 API
- V1 backend and V2 API server run in parallel with no interference (frontend toggle verified)
